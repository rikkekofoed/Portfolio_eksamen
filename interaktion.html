HTML

HTML står for Hyper Text Markup Language og det er det "sprog" man skriver hjemmesider i, når man sætter den basale struktur op.

Det er i HTML at man bygger en hjemmesides skelet op med f.eks. navigationsbar, header, footer, overskrifter og sections.

HTML består overordnet set af to dele: en head og en body. I head'en (ikke at forveksle med en header) placeres alt det man ikke ser når man besøger en webside.
Det gælder f.eks. metatags og links til stylesheets osv.
I body'en placeres alt indholdet så, altså alt hvad man umiddelbart ser på selve hjemmesiden, skal placeres her. Det gælder for alt tekst, billeder og f.eks. også en navbar.

CSS

CSS står for Cascading Style Sheets og det er det "sprog" man bruger til at style sin HTML og altså bestemme hvordan skelettet ser ud.

Hvis man tænker på HTML som skelettet på et menneske, vil CSS være huden på mennesket. Det er i CSS man bestemmer skrifttypen, farverne, skriftstørrelsen og meget mere.
Det er også i CSS at man kan positionere sit indhold, f.eks. i form af Flexbox eller CSS Grids.
Den måde vi primært har arbejdet med positionering af indhold er gennem CSS Grids, som jeg kommer ind på senere.

Man arbejder typisk med CSS i et seperat dokument ift. sit HTML-dokument, og man beder HTML-dokumentet om at hente CSS-filen (det sker i head).
Det er også muligt at bruge CSS direkte i sin HTML-fil, ved at bruge style tagget i head sektionen, eller man kan bruge inline styling, hvor man styler direkte på et tag.
De sidste to muligheder skal bruges så lidt som muligt, da man ellers blander strukturen fra HTML'en sammen med stylingen fra CSS.

Layout strategier

Statisk layout

Det statiske layout var dominerende i de tidlige år i www's historie. Her baserede man sit layout på faste mål, og derfor fungerede layoutet typisk ikke på andre skærmstørrelser end den tiltænkte (i hvert fald ikke særlig godt.)
Da layoutet ikke ændrede sig efter skærmstørrelser, ville man se luft på en for stor skørm, mens der blev zoomet på en mindre skærm.
Man bruger ikke statisk layout længere, da man i dag arbejder med mange skærmtyper.

Fluid webdesign

Her arbejder man med elementer der tilpasser sig den bredde der er på skærmen. Man angiver ikke bredder i faste mål, men de angives i relative mål f.eks. % som er meget udbredt.
Det giver en bedre brugeroplevelse end den statiske tilgang, da indholdet som sagt tilpasser sig.
Fluid webdesign bliver ofte brugt når indholdet og funktionaliteten er mere vigtigt end designet.

Adaptivt layout

Den adaptive tilgang bruges mest hvis man har en række statiske layouts som man gerne vil gøre nogenlunde responsive uden at skrive alt om.
Der er her tale om en mellemting mellem et statisk layout og et fuldt responsivt layout.
Det adaptive layout består af en række statiske layouts, som styres af @mediaqueries.
Rammer en skærmstørrelse ikke størrelsen på breakpointet, kommer problematikken med for meget eller for lidt plads igen, ligesom med det statiske layout.
Man bruger ikke rigtig det adaptive layout hvis man har muligheden for at gøre det responsivt.

Responsivt layout

Det responsive layout er dagens standard, fordi det er forholdsvis nemt at have med at gøre samtidig med at det passer på langt de fleste skærmstørrelser i dag.
Det består nemlig typisk af fluid webdesign (med de relative bredder) og derudover nogle regler, som ændrer layoutet efter skærmstørrelse.
Det vil sige, at elementerne tilpasser sig bredden hvis skærmstørrelsen ikke rammer breakpointet helt præcist.
Reglerne bliver styret af @media

Versionsstyring

Man bruger versionsstyring til at sikre, at man altid kan gå tilbage til en anden version, hvis der skulle ske en fejl med det projekt man er i gang med.
Det kan være meget simpelt at versionsstyre ved f.eks. navngivning af mapper og filer med dertilhørende noter omkring ændringerne. Man kan altså gøre det manuelt.
Langt de fleste vælger dog at bruge et Version Control System (VCS), hvor der er en software der automatisk går ind og holder styr på forskellene mellem versioner.
En af de meget udbredte systemer, og den vi er blevet introduceret til i undervisningen, er Git-protekollen, der altså er et versionsstyrings system.
Det er muligt selv at oprette en Git-server, men man kan også sagtens bruge en online tjeneste som f.eks. Github.
Github er den tjeneste vi er blevet introduceret til, og som vi bruger til f.eks. vores projekter.

Github

GitHub fungerer som en online server til versionsstyring med Git - så behøver man ikke selv oprette en.
GitHub er blevet meget udbredt blandt webudviklere, og bruges i dag også som en slags portfolio for ens kode.
Github kan bruges via kommandolinjen, eller som langt de fleste vælger via GitHub Desktop.

GitHub fungerer ved, at man opretter et repository som er en slags projektmappe, hvori alle ens filer for projektet ligger.
Efterhånden som man får lavet noget på sit projekt committer man, og det fungerer ligesom hvis man gemmer en fil på traditionel vis
med den funktionalitet at man kan gå tilbage til tidligere versioner.
Det kan også være en god idé at lave en ny branch som er en slags parallel version af ens master branch som er den branch der gælder.
På den nye branch kan man altså lave ændringer, uden at de berører ens master.
Når man er ved at være ved vejs ende kan man lave en pull request hvor man egentlig vil have forslag til en ny version, med rettelser og alt det der.
Der kan man også se hvilke diffs der er, hvilken betyder hvilke forskelle der er mellem to commits.
Når man er tilfreds med det kan man merge det hele sammen, og det betyder at man samler ændringer i én branch med en anden branch.

GitHub gør det også ret nemt at arbejde flere sammen om et projekt, netop på grund af den måde den versionsstyrer på.
Vi bruger som sagt GitHub når vi laver projekter, hvilket vil sige, at koden til denne portfolio også er at finde på GitHub.

CSS Grid Model

En splinterny måde at positionere sit indhold i CSS. Der har aldrig været en rigtig god måde at gøre dette på, og derfor har de eksisterende funktioner ofte været brugt til mere end de burde.
F.eks. fungerer Flexbox kun i en dimension, hvor CSS grid fungerer i to dimensioner (både vertikalt og horisontalt).
Det er super nemt at arbejde med, og er et skridt på vejen til at skabe responsive websites rimelig nemt.
Det er som sagt meget nyt, og derfor er der meget jeg overhovedet ikke har lært endnu, men det basale er at få sat det op.
Man kan også sagtens have grids inden i et grid, og det er på ingen måde besværligt at bruge.

Man sætter bare en display:grid op i den container man gerne vil have (f.eks. en pagewrap), og derefter sætter man grid-template-colums og grid-template-rows for at definere hvor mage kolonner og rækker man gerne vil have.
Brug fr (fractions) som betyder dele, hvilket gør det nemt at skalere det op og ned i skærmstørrelser.

Der er så forskellige måder at sætte det op på, men grid-area er det jeg synes er nemmest.
Man giver hver del et grid-area og så der hvor man har det andet laver man en grid-template-areas hvor man angiver de forskellige areas alt efter hvordan man vil have det til at se ud.

Frameworks

Det er en samling af standardiserede kodeelementer som man kan bruge til at opbygge layout og funktionalitet hurtigt og effektivt.
Man slipper for at lave alt koden fra bunden, fordi man kan hente elementer ind, der allerede er lavet.
Du skal heller ikke starte forfra hver gang, du koder en hjemmeside.

Fordelene ved at anvende frameworks er, at det bl.a. gør det nemmere at gøre en side brugervenligt, samtidig med at det som sagt går en del hurtigere at lave layout.
Ulemperne kan være, at hjemmesider der er lavet på framworks meget ofte har nogle fællestræk, og der kan derfor være svært at lave et unikt design, når du bruger framworks (det er i hvert fald vanskeligere).

Vi er blevet introduceret til Bootstrap, som nok er det mest populære framework for både HTML, CSS og Javascript, og som også har den fordel at den er responsiv (langt det meste i hvert fald).
